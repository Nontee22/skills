# 优化文章表达

## 概述

对用户提供的文章进行表达优化和排版重构，使文章结构清晰、层次分明，适合新手阅读学习。**核心铁律：文章内容绝对不允许缺失，一字一句都不能丢。**

## 触发条件

- 用户说"优化这篇文章""优化表达""优化排版"
- 用户说"让文章更易读""适合新手""整理格式"
- 用户说"文章润色""重新排版""改进可读性"
- 用户提供一篇文章并要求"帮我改改"

## 核心铁律（违反即失败）

### 规则零：内容绝对不缺失

**这是最高优先级规则，覆盖所有其他规则。**

- 原文的每一个知识点、每一个概念、每一个数据、每一个例子，都必须出现在优化后的文章中
- 不允许删除任何一句话、一个词，除非是纯语气词（如"呢""吧""嘛"且删除后不影响语义）
- 不允许"概括""精简""合并"导致信息丢失
- 如果原文有 10 个要点，优化后必须有 10 个要点，不能变成 8 个
- 优化完成后，必须逐段对照原文，确认没有遗漏

### 规则一：不能做的事

- 不增：不在原文没有提及的地方添加新知识、新观点、新数据
- 不减：不删除任何实质性内容（知识点、概念、例子、数据、步骤）
- 不改意：不改变原文的任何结论、观点、立场
- 不概括：不把详细说明压缩成一句话
- 不翻译：原文是什么语言，输出就是什么语言

### 规则二：可以做的事

- 调整段落顺序，使逻辑更流畅（但不丢失任何段落）
- 添加小标题和编号，建立层次结构
- 拆分过长的段落（>15 行），每个子段落聚焦一个点
- 合并过短的段落（1-2 句且同一主题），但要保留所有句子
- 添加"关键点""小结""注意"等提示框，但不新增内容
- 调整标点符号和连接词，使句子更通顺
- 将长句拆分为短句，将隐含的逻辑关系显式化（如加上"因为""所以""首先""其次"）

## 执行流程

```
用户提供文章
      │
      ▼
步骤一：通读分析（MANDATORY）
- 通读全文，标记所有知识点、概念、例子、数据
- 识别文章的核心主题和子主题
- 输出知识点清单（用于后续对比验证）
      │
      ▼
步骤二：结构重组
- 按"总-分-总"或"递进式"逻辑重新组织段落
- 为每个子主题添加小标题
- 确定层级关系（一级标题、二级标题、三级标题）
      │
      ▼
步骤三：表达优化
- 拆分长句（超过 40 字的句子拆成 2-3 句）
- 拆分长段落（超过 15 行的段落按主题拆分）
- 添加过渡句和逻辑连接词
- 关键概念加粗或高亮（用 **粗体**）
- 对需要强调的规则/原则使用引用块（> ）
      │
      ▼
步骤四：排版增强
- 列表化：将并列内容转为无序列表或有序列表
- 代码/术语用行内代码块（` `）
- 步骤类内容用有序列表
- 对比类内容用表格（如有需要）
- 添加分隔线区分大板块
      │
      ▼
步骤五：内容完整性验证（MANDATORY）
- 逐条对照步骤一的知识点清单
- 确认每一个知识点都在优化后的文章中出现
- 确认没有新增原文不存在的内容
- 确认没有改变原文的任何结论
      │
      ▼
输出优化后的文章为 .md 文件 + 优化说明
```

## 步骤一：通读分析

**MANDATORY。不可跳过。**

1. 仔细阅读用户提供的文章全文
2. 提取并记录以下内容清单：
   - **知识点**：文章中教给读者的每一个概念、原理、方法
   - **关键数据**：所有数字、日期、参数、配置值
   - **例子/案例**：所有举例和示范
   - **步骤**：所有操作步骤或流程
   - **结论/观点**：所有判断、建议、总结
3. 识别文章的主题结构（总分结构 / 递进结构 / 并列结构）

输出知识点清单格式：
```
知识点清单：
1. [知识点1描述]
2. [知识点2描述]
...
N. [知识点N描述]

关键数据：X个
例子：X个
步骤：X个
结论：X个
```

## 步骤二：结构重组

### 标题层级规范

- 全文只用一个一级标题（`#`），即文章主标题
- 主标题下用二级标题（`##`）划分大板块
- 二级标题下用三级标题（`###`）划分子主题
- 不再使用更深层级（四级及以上），新手容易迷失

### 段落拆分与合并

**拆分条件（满足任一条即拆分）：**
- 段落超过 15 行
- 一个段落包含 2 个及以上不同主题
- 段落中有步骤，且步骤 ≥ 3 步

**合并条件（同时满足才合并）：**
- 两个相邻段落 ≤ 2 句
- 属于同一主题
- 合并后不超过 10 行

### 标题命名原则

- 标题必须是完整的短语，而不是单个名词
- 标题要让读者一眼看出这段讲什么
- 好："如何配置数据库连接" / 差："数据库"
- 好："常见错误及解决方法" / 差："错误"

## 步骤三：表达优化

### 句子优化规则

1. **长句拆分**：超过 40 字的句子，拆成 2-3 个短句
2. **逻辑显式化**：原文隐含的因果、转折、递进关系，用连接词显式标出
   - 因果关系 → 加上"因为……所以……""因此""由于"
   - 转折关系 → 加上"但是""然而""不过"
   - 递进关系 → 加上"此外""不仅如此""更重要的是"
3. **主语补全**：原文省略的主语，补全为明确的主语
4. **被动转主动**：非必要的被动语态转为主动语态
5. **术语首次出现加说明**：如果术语对新手不友好，在首次出现时用括号加一句简短说明

### 重点标注规则

- **粗体**（`**文字**`）：用于关键概念、核心结论、重要提醒
- 行内代码（`` `文字` ``）：用于代码、命令、文件名、参数名
- 引用块（`>`）：用于需要特别注意的规则、警告、铁律
- 每 300 字最多使用 3 处粗体标注，避免到处都是重点

## 步骤四：排版增强

### 列表使用

- 并列关系 → 无序列表（`-`）
- 顺序关系/步骤 → 有序列表（`1.`）
- 列表项如果是完整句子，末尾加句号；如果是短语，不加标点

### 表格使用

仅在以下情况使用表格：
- A 与 B 的对比（如"优点 vs 缺点""方法A vs 方法B"）
- 参数/配置汇总
- 表格不超过 5 列、10 行，超过则拆分

### 代码块使用

- 多行代码 → 围栏代码块（`` ``` ``）并标注语言
- 单行代码/命令/参数 → 行内代码（`` ` ``）

### 提示框

以下情况使用提示框格式：
```
> **关键点**：[一句话核心要点]
> **注意**：[需要特别小心的内容]
> **提示**：[实用小技巧]
```

提示框内的内容必须是原文已有的信息，不能新增。

## 步骤五：内容完整性验证

**MANDATORY。这是最后一道关卡，不可跳过。**

验证方法：
1. 拿出步骤一记录的知识点清单
2. 在优化后的文章中逐条搜索，确认每条都出现
3. 用原文和优化后的文章做段落级比对：原文有 N 段实质性内容，优化后也有 N 段
4. 确认没有新增原文不存在的信息
5. 确认结论、观点与原文完全一致

如果发现缺失 → 回到缺失位置，补回内容
如果发现新增 → 删除新增内容
如果发现结论改变 → 修正回原文立场

通过验证后才可输出。

## 输出规范

优化后的文章必须以 Markdown 文件（`.md`）形式保存到工作目录。

### 文件命名

- 如果用户未指定文件名：在原文文件名后加 `_优化版`，如 `文章名.md` → `文章名_优化版.md`
- 如果用户没有提供原文件名：根据文章主题自动生成一个有意义的文件名，如 `Python入门指南_优化版.md`
- 文件名只使用中文、英文、数字和连字符，不使用特殊符号

### 文件内容格式

输出的 `.md` 文件内容直接是优化后的文章正文，使用标准 Markdown 语法：

```markdown
# 文章主标题

## 第一个大板块

正文内容……

### 子主题

正文内容……

## 第二个大板块

正文内容……
```

### 输出时必须同时提供

1. 将优化后的文章写入 `.md` 文件
2. 向用户报告优化说明，包括：
   - 优化了哪些方面（结构、表达、排版各做了哪些改动）
   - 确认内容完整性（知识点数量比对：原文 X 个 → 优化后 X 个）
   - 输出文件路径

## 失败处理

以下情况必须停止并报告用户：
- 原文内容有歧义，无法确定作者的准确意思 → 列出歧义处，请用户澄清
- 文章主题超出知识范围，无法判断哪些是核心内容 → 说明原因，请用户确认知识点
- 原文中存在明显的自相矛盾 → 指出矛盾处，请用户确认正确版本

不要在不确定的情况下自行判断或猜测。
